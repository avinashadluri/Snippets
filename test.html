Adoption of WSSO Proxy Farm

IAM has an established reverse proxy farm for both intranet and internet WSSO standard integrations. The reverse proxy farm offers some advantages and is the default integration method for WSSO standard integrations. Please review the following and determine if your application is suitable for the WSSO proxy farm solution.
  • Lowers KP's overall operational cost. Application owners do not install, configure, or maintain the WebGate component. The WebGate component is centrally managed by the WSSO team including the installation, configuration, maintenance (e.g. certification renewal). Application owners are responsible for maintaining the connectivity and security of their application between the WSSO proxy farm servers and their application (e.g. load balanced URL, two-way certificates).
  • Application owners may not need to work with both the WSSO team and IBM Web Support when there is an issue related to WSSO.
  • The WSSO reverse proxy farm is highly available with servers geographically distributed across multiple data centers.
  • If your application is not using both a certified KP and Oracle IAM web server and/or web application server, the WSSO proxy farm may be a good alternative. A certified KP and Oracle IAM web server and/or application server combination is not required for most WSSO proxy farm integrations.
  • However, there are some additional requirements and caveats:
    • The application must use the standard WSSO proxy configuration, such as a maximum session timeout of nine (9) hours, because the configuration is shared among many applications.
    • Unless the application resides as a single instance on a server, a load balanced backend URL to the application is required.
    • Applications must secure the connection between the WSSO farm proxy servers and the application's servers. The application's servers need to disallow bypass of the WSSO proxy servers. Traffic to the application's servers must be limited to only go between the proxy farm and your web/application servers. The application's servers should not be directly accessible.
    • The WSSO proxy farm does not support the use of specific web server / web application plugins (e.g. WebSphere plugin). If your application requires a plugin, then an additional web server between the reverse proxy server farm and the application server is needed.
    • A preferred step of the integration with the WSSO proxy farm is to re-point the application's existing DNS (appsubdomain.kp.org) to the proxy farm to keep using the existing application URLs (e.g. no changes to a user's internet browser bookmarks). However, there may be some limitations if multiple applications are hosted within your application's subdomain and are not all protected by WSSO. For example, mysubdomain.kp.org contains multiple applications:
      • App1 with a URL of appsubdomain.kp.org/app1 and
      • App2 with a URL of appsubdomain.kp.org/app2.
      Only app1 will be protected by WSSO and App2 will not be protected by WSSO. The DNS cannot be split to support both applications. The scenario above is rare for most KP applications. However, if the scenario does apply to your application (e.g. App1), there is an alternative option. The alternative option will use a new subdomain and URL for the application to avoid a conflict with other applications on the same existing subdomain. The application URLs will be different as a result (e.g. newappsubdomain.kp.org/app1)".
    • If the application's existing end-user URL cannot be allocated to the WSSO proxy farm (e.g. due to technical challenges), there is an alternative option which uses a new subdomain as the end-user's URL. The application's end-user URLs will be different as a result (e.g. appsubdomain-appl.kp.org). This allows the existing URL to be used as the backend URL for the application.

WSSO Integration Agreement

Download WSSO Integration Process and Responsibilities
Changes to your WSSO integrated application may impact the WSSO infrastructure, WSSO related systems (e.g. Enterprise Directory), WSSO support, and/or other WSSO integrated applications. Thus, you must agree with the following terms before we can proceed with the integration.
  • It is your responsibility to notify us if the application owner and/or application owner contact information changes. There are many occasions that we need to notify application owners about upgrades, patches and outages.
  • The application owner must provide sufficient advance notice of the following changes:
    • Decommission of the application.
    • Significant change in the user population and/or peak concurrent user sessions
    • Web Server change/upgrade
    • Application Server change/upgrade
    • OS change/upgrade
    • Removal and/or addition of servers
    • Relocation of the servers hosting your application
    • Major application upgrade or changes
    • Major application architecture change
    • DNS change
    • Load Balancer change
  • It is the responsibility of the application owner to ensure the application and its users do not exceed the estimated user usages (e.g. peak user population, peak concurrent user usage, etc.) provided earlier. Performance and stability of your application may be affected. In addition, exceeding the estimated user usages may severely impact the overall stability of WSSO and all other integrated applications. The application owner may be held responsible for any WSSO and application outages that occur due to exceeding the estimated application user usage.
  • WSSO and/or its components (e.g. WebGate) may be upgraded and/or changed in the future. You must use the supported version of WSSO and/or its components. We will notify the application owner when testing, validation, installation, upgrade, patch, or a configuration change is required. The application owner agrees to manage the work and complete it within a reasonable time as requested by the TRO. (Example 1: The proxy farm's WebGate may be updated; the application owner is responsible for testing of their application. Example 2: A quarterly security patch is released; the application owner is responsible for applying the quarterly patch and testing their application)
  • The WebGate ID must be maintained for its original purpose and environment and should not be shared with other applications/environments due to serious implications.
  • The certificate renewal and maintenance of WebGate are the application owner's responsibility. However, WSSO team can provide help if needed. The application owner will be notified when the certificate is near expiration.
  • Please be sure that you have reviewed and agreed to the WSSO integration process, responsibilities, requirements, and guidelines (as linked above). For additional information about WSSO, visit: https://wiki.kp.org/wiki/display/ca/Web Single Sign On (WSSO)
  • Please note: All WSSO clients should subscribe to the "WSSO NotifyAll-IREG" email distribution list through Lotus Notes. This group will receive important announcements regarding WSSO environment outages etc. It is the client's responsibility to subscribe to "WSSO NotifyAll-IREG", and to keep their contacts up-to-date.
